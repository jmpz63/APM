§0Phantom Typist

A non-invasive robotic overlay that sits on top of your existing keyboard and beside your mouse, and physically presses keys, moves the mouse, clicks, and scrolls — driven by commands from your AI. Nothing is wired into the target computer; the rig acts through the same mechanical interface a human hand would, so the target machine cannot tell software from a "person."

Non-invasive overlay 104-key solenoid array CoreXY mouse gantry ESP32-S3 · WiFi/BLE AI over WebSocket ~$430 BOM
Read this first — engineering reality check. If your goal is simply "let my AI type and click," a USB HID emulator (a $6 microcontroller pretending to be a keyboard/mouse) does the same job for 1% of the cost, instantly, with zero mechanics. You already have a Bluetooth_HID_Emulator project in this workspace that does exactly that.

The only reasons to build a physical actuator rig instead: (1) the target machine is air-gapped / locked-down and rejects new USB devices; (2) you need to demonstrate genuine physical presence (defeating "is-a-human-at-the-desk" sensors, kiosk lockdowns); (3) it's a showpiece / art / robotics learning build. This document designs the physical rig — but be honest with yourself about which problem you actually have.
Responsible-use note. This is designed for automating your own machine. Using it to defeat presence checks, proctoring, or access controls on systems you don't own may violate terms of service or law. Build it for your own desk.

What it does

Presses any of 104 keys individually or in combos (Shift/Ctrl/Alt), moves the mouse to XY targets, left/right clicks, and scrolls — all on real hardware.

How it's driven

Your AI (on forge1 or any host) sends high-level commands — type "hello", moveTo(x,y), click — over WiFi to the on-rig ESP32-S3, which sequences the actuators.

Why an overlay

It clamps over your existing keyboard with alignment feet — no soldering to your keyboard, no modification. Lift it off and your desk is normal again.

§1System Architecture

Three domains: the AI/host (decides what to type), the controller (translates intent into actuator timing), and the actuator hardware (does the physical work). The target computer is a passive recipient — it only sees keypresses and mouse motion.

§2Keyboard Actuator Overlay

A rigid plate suspended a few millimetres above the keycaps on adjustable feet. Each key location carries a small push-pull solenoid whose plunger tip lands on the centre of that keycap. Energising the solenoid drives the plunger down ~4 mm — enough to actuate the switch — then a return spring lifts it.

Fig 2 — One of 104 identical actuator stations. The silicone tip protects keycaps and adds compliance so exact height isn't critical.

Addressing 104 solenoids without 104 GPIO

The ESP32 can't drive 104 lines directly, so a daisy-chained shift-register bus does it over 3 wires. 13× 74HC595 (8 outputs each = 104) are clocked via SPI. Each output feeds one channel of a ULN2803A Darlington array, which sinks the solenoid current and includes the flyback clamp diode internally. To type a character the controller shifts out a 104-bit frame with the target bit(s) set, latches it, waits the pulse time, then clears it.

Combos are free. Because every key has its own driver, Ctrl+Shift+Esc is just three bits set in the same frame — no special-casing. The only constraint is the power budget: hold no more than ~4 solenoids at once (see §5 power math).

Plate & frame

3 mm laser-cut acrylic or aluminium key-plate with a hole pattern matching a standard ANSI 104 layout. Mounted in a 2020-extrusion perimeter with four thumb-screw feet that rest on the desk around the keyboard, not on it.

Alignment

Two locating pins drop into the gaps beside F1 and the numpad Enter. A one-time calibration wizard lights each solenoid so you can confirm tip-to-keycap registration, then saves offsets.

§3Mouse Actuator Gantry

Rather than emulate the mouse electrically, the rig drives your real mouse on a CoreXY gantry — the mouse rides in a printed cradle, so its own sensor reports true motion to the computer. Two NEMA-17 steppers fixed to the frame move the cradle in X and Y via a single GT2 belt loop (CoreXY). Click solenoids press the physical buttons; a servo-driven friction wheel turns the scroll wheel.

Fig 3 — CoreXY keeps both heavy motors on the frame; only the light cradle moves, allowing fast, accurate positioning. Homing via two optical endstops (not shown) sets the origin.

Why CoreXY over a mouse-emulator

The computer's pointer acceleration, sensor DPI, and surface behaviour are all preserved because the actual optical sensor is doing the sensing. The AI issues target pixels; a small closed-loop routine nudges and reads back cursor position via the host (optional) or dead-reckons from a homed origin + known DPI.

Clicks & scroll

Two 5 V pull solenoids sit over the L/R buttons. Scroll uses an SG90 micro-servo turning a rubber friction wheel resting on the mouse's scroll wheel — discrete steps map to scroll "notches."

Cheaper alternative for the mouse: skip the gantry entirely and let the AI send relative mouse deltas through the same USB-HID path as the keyboard fallback (§0). Keep the gantry only if physical presence is the point.

§8Firmware & Command Protocol

The ESP32 runs a WebSocket server. Your AI connects and streams JSON commands; the firmware queues and executes them with power-aware scheduling.

Command protocol (JSON over WS)

CommandEffect
{"type":"key","chars":"Hello!"}types the string, auto-shifting
{"type":"combo","keys":["ctrl","c"]}simultaneous press/release
{"type":"move","x":842,"y":391}gantry to pixel target
{"type":"click","btn":"left"}click solenoid pulse
{"type":"scroll","notches":-3}servo rotates wheel
{"type":"home"}re-home mouse origin

Execution rules

  • Power gate: never energise >4 key solenoids; queue overflow keys to next slot.
  • Debounce: 30–50 ms press, 20 ms release before next key.
  • Mouse/key mutex: steppers pause during key bursts to protect the PSU.
  • Homing: optical endstops set origin at boot; positions dead-reckoned from DPI.
  • Watchdog: lost WS link → release all actuators (fail-safe open).

§9Build Sequence

  1. Prototype one channel first. ESP32 + one 595 + one ULN + one solenoid. Prove SPI shift, latch, and a clean key press before scaling to 104.
  2. Build the driver PCB / protoboard — 13× (595+ULN) banks, shared SPI bus, power rails, bulk cap, fuse.
  3. Fabricate the key-plate from your keyboard's exact layout (measure or use the ANSI-104 DXF).
  4. Populate solenoids into printed holders; fit silicone tips; wire in banks of 8 to each ULN.
  5. Run the calibration wizard — light each key in sequence, adjust feet until every tip actuates reliably.
  6. Assemble the CoreXY mouse gantry; home it; verify pixel-accurate moves with a test pattern.
  7. Integrate the WebSocket link to your AI; test type, combo, move, click.
  8. Full-system soak test — type a known paragraph, diff the result on-screen; measure WPM and error rate.
Verification mirrors the FORGE QA discipline: define the pass condition first (e.g. "types this 500-char paragraph with zero errors at ≥60 WPM"), test the real hardware, and turn every mis-press into a calibration regression case.
